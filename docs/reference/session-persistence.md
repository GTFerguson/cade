# Session Persistence for Agentic Development Environments

> [!NOTE]
> This document grounds session persistence patterns in evidence. See [[self-improving-agent-systems]] for agent memory and [[agentic-context-engineering]] for context management.

## Overview

Session persistence enables agentic development environments to survive disconnections, resume interrupted work, and maintain state across sessions. This document analyzes state models, serialization approaches, recovery patterns, PTY persistence, and security considerations based on current research and implementations.

The core tension: agents need sufficient state to continue meaningful work, but persisting sensitive data creates security risks and storage overhead.

## 1. Session State Models

### 1.1 Session vs Context vs History

Modern LLM agent architectures distinguish between three interconnected but distinct state constructs:

| Construct | Temporal Scope | Volatility | Content |
|-----------|---------------|------------|---------|
| **Working Memory** | Current task/session | In-memory | Active context window, immediate reasoning state |
| **Episodic Memory** | Historical experiences | Persistent | Concrete experiences, tool traces, partial results |
| **Semantic Memory** | Accumulated knowledge | Persistent | Abstracted facts, project conventions, learned patterns |
| **Procedural Memory** | Reusable capabilities | Persistent | Skills, plans, executable code fragments |

Research distinguishes these from related concepts:

- **Session**: A discrete interaction period, bounded by connect/disconnect events
- **Context**: The bounded input provided to the LLM at inference time (finite token window)
- **History**: The chronological record of interactions within or across sessions

Agent memory differs from LLM memory (which manages transformer KV cache) and from RAG (which augments individual inference with static external knowledge). Agent memory operates within the agent's ongoing environmental interaction, continuously incorporating new information generated by the agent's actions (Memory in the Age of AI Agents, 2025).

### 1.2 Claude Code Resume Architecture

Claude Code implements session persistence through the `--resume` flag, which loads a saved session file to continue interrupted work. The architecture stores:

- Complete conversation history with all user prompts and assistant responses
- Tool call history including arguments, results, and any errors
- Working directory state (file modifications pending)
- Execution context for long-running processes

Session files are stored locally (typically `~/.claude/sessions/`) and include metadata for session identification and timestamp. The resume mechanism reconstructs the agent's reasoning context by replaying history and re-establishing the task state.

## 2. Context Serialization

### 2.1 State Components

Agent session state comprises multiple layers requiring serialization:

**Primary State:**
- Chat history (message序列 with roles, content, metadata)
- Tool call history (function invocations, parameters, outputs)
- Working memory (current reasoning state, active plans)

**Secondary State:**
- File system state (workspace contents, modification timestamps)
- Environment variables and configuration
- Network connections and API session state

**Derived State:**
- Vector indices for retrieval-augmented memory
- Summarized or compressed history (for long sessions)
- Execution checkpoints for resumable tasks

InfiAgent (2026) externalizes persistent task state into a file system workspace, treating files as the authoritative record. This "file-centric state abstraction" separates bounded reasoning context from unbounded task history, enabling stability over arbitrarily long horizons.

### 2.2 Serialization Approaches

| Approach | Mechanism | Tradeoff |
|---------|-----------|----------|
| **Full serialization** | Dump entire state to disk (JSON, msgpack) | Complete restore fidelity, large file sizes, slow I/O |
| **Incremental checkpointing** | Periodic snapshots + diff logs | Efficient storage, potential state gaps on recovery |
| **Delta compression** | Store only changes from baseline | Balances efficiency and completeness, complexity in diff resolution |
| **Summarization-based** | Compress history via LLM summarization | Constant context size, information loss over time |
| **Tiered memory** | Hot/warm/cold storage tiers (OS-inspired paging) | Adaptive cost/performance, orchestration overhead |

Context-resident memory (sliding windows, rolling summaries) suffers from "summarization drift" (loss of detail) and "attentional dilution" (degraded focus in longer contexts), making it insufficient for long-running agents. Hierarchical memory systems like MemGPT address this through explicit tier management.

## 3. Session Recovery and Resume

### 3.1 Crash Recovery Patterns

Robust session recovery requires handling multiple failure modes:

**Connection Loss:**
- Detect disconnection via heartbeat timeouts or explicit signals
- Preserve in-flight state before connection terminates
- Reconnect to persisted session rather than spawning new context

**Process Crash:**
- State externalized before crash (write-ahead logging)
- Recovery from last confirmed checkpoint
- Rollback of uncommitted changes

**Platform Migration:**
- Session portability across different client instances
- Authentication re-establishment without state loss

The "write–manage–read loop" formalizes this: memory update functions process current state, actions, observations, and rewards to maintain the agent's belief state (Memory for Autonomous LLM Agents, 2026). This POMDP-aligned view treats memory as a sufficient statistic of history subject to computational and storage constraints.

### 3.2 Checkpoint Granularity

Checkpoint timing significantly impacts recovery fidelity vs. storage overhead:

| Strategy | Trigger | Storage Cost | Recovery Fidelity |
|----------|---------|--------------|-------------------|
| **Time-based** | Fixed intervals (30s, 5m, 1h) | Predictable growth | Depends on interval length |
| **Action-based** | After each tool call or significant action | Higher frequency | Fine-grained |
| **Milestone-based** | After task completion, plan acceptance | Minimal | May miss intermediate progress |
| **Adaptive** | Combines time + action + importance scoring | Optimized | Requires sophisticated logic |

InfiAgent implements "Periodic State Consolidation," where a fixed-length buffer of recent actions reconstructs reasoning context by querying the persistent workspace. This "zero context compression" approach ensures bounded context while retaining full task history externally.

### 3.3 Long-Running Tasks

Interrupted long-running tasks present unique recovery challenges:

**State Reconstruction:**
- Identify task boundaries and current progress point
- Reconstruct active plans, subgoals, and pending dependencies
- Validate workspace consistency (files may have changed)

**Error Recovery:**
- Distinguish partial completion from failed steps
- Determine retry strategy (resume vs. restart)
- Handle side effects from partially-executed steps

**Human-in-the-Loop:**
- Notify user of interruption and recovery options
- Present recovery summary before auto-resume
- Allow user to modify plan before continuation

Agents with effective memory mechanisms demonstrate significant performance degradation when memory is removed, often more pronounced than differences from varying LLM backbones (MemoryArena findings).

## 4. PTY Session Persistence

### 4.1 Terminal Session Architecture

Pseudo-terminal (PTY) sessions maintain interactive shell state separate from the agent's primary session:

**tmux/screen:**
- Server-client architecture with named sessions
- State stored in sockets and Unix domain sockets
- Detach/reattach workflow for persistence
- Panes, windows, and layout state preserved

**VS Code Terminal:**
- Terminal processes linked to workspace
- State serialized to extension host
- Reconnect via terminal restoration API

**SSH Sessions:**
- Network-level persistence via SSH keepalives
- Terminal state transmitted over encrypted channel
- Reconnection requires new PTY allocation

### 4.2 Reconnection Patterns

| Pattern | Mechanism | Limitations |
|---------|-----------|-------------|
| **Socket-based** | Reattach to existing socket (tmux, screen) | Same user, same socket path |
| **Proxy-based** | Terminal proxy maintains state, clients reconnect | Additional infrastructure, latency |
| **Snapshot-based** | Capture terminal state, replay on reconnect | Scrollback loss, process state may diverge |
| **Native reconnect** | Server supports session handoff | Protocol-specific, limited adoption |

tmux provides robust persistence: sessions survive client disconnection, state is maintained in server memory, and clients reattach via `tmux attach`. The trade-off is memory usage for long-running sessions with large scrollback buffers.

### 4.3 State That Survives Reconnect

**Preserved:**
- Shell history (bash/zsh history files)
- Environment variables and aliases
- Current working directory
- Running processes (background jobs, interactive programs)
- tmux/screen windows, panes, and layouts

**Lost or Reset:**
- Scrollback buffer (unless explicitly configured)
- Local shell variables (non-exported)
- Terminal dimensions (may reset on reconnect)
- Interactive program state (depends on program)

**Configurable:**
- Scrollback buffer size (tmux `history-limit`, screen `logfile flush`)
- History deduplication and timestamping
- Session naming for targeted reconnection

## 5. Security Considerations

### 5.1 Risks of Persisted Session State

Session persistence introduces significant attack surface:

**Credential Exposure:**
- API keys, tokens, and secrets may reside in environment variables or history
- SSH keys, database passwords, cloud credentials in shell state
- Authentication tokens cached for convenience

**Data Leakage:**
- File contents processed by the agent may be cached
- Query parameters and API responses in tool call history
- Project-specific knowledge (trade secrets, unreleased code)

**Privilege Escalation:**
- Persisted sessions may retain elevated permissions
- Stale tokens may enable unauthorized access
- Session files inherit filesystem permissions

**Lateral Movement:**
- Compromised session files enable session hijacking
- Shared workspaces leak state between users
- Backup copies may not inherit security policies

The practical memory injection attack demonstrates that agents with persistent memory are vulnerable to malicious memory corruption, where adversaries can manipulate stored information to influence agent behavior (Memory Injection Attack, 2025).

### 5.2 Mitigations

| Mitigation | Approach | Tradeoff |
|-----------|----------|----------|
| **Encryption at rest** | Encrypt session files with per-user keys | Key management overhead, CPU cost |
| **Filesystem permissions** | Restrict session directory to owner | Prevents multi-user sharing |
| **Time-to-live (TTL)** | Auto-expire sessions after inactivity | Loss of long-term context benefits |
| **Secrets exclusion** | Strip credentials before serialization | May break functionality requiring secrets |
| **Memory isolation** | Separate sensitive from non-sensitive state | Architectural complexity |
| **Audit logging** | Track session access and modifications | Storage overhead, privacy implications |
| **Hardware-backed keys** | TPM, secure enclave for encryption keys | Platform dependency |

For agentic environments, three architectural patterns emerge: monolithic context (simple, limited scale), context + retrieval store (common, workhorse), and tiered memory with learned control (sophisticated, highest potential). Each trades security isolation against complexity and performance.

Memory governance—including privacy, deletion, and compliance—represents a key design objective that must be balanced against utility, efficiency, adaptivity, and faithfulness (Memory for Autonomous LLM Agents, 2026).

## 6. Evidence Summary

| Pattern | Tier | Source |
|---------|------|--------|
| Hierarchical memory (working/episodic/semantic/procedural) | Strong | Memory in the Age of AI Agents (2025) |
| File-centric state externalization for long-horizon stability | Strong | InfiAgent (2026) |
| Context-resident memory suffers summarization drift | Strong | Memory for Autonomous LLM Agents (2026) |
| Checkpoint granularity impacts recovery fidelity | Moderate | Engineering practice (tmux, screen, IDEs) |
| PTY state preservation via socket-based reconnection | Strong | tmux/screen architecture documentation |
| Secrets in persisted state create credential exposure risk | Strong | Memory Injection Attack (2025) |
| Memory removal causes more degradation than LLM backbone differences | Strong | MemoryArena benchmark |
| Tiered memory (hot/warm/cold) inspired by OS virtual memory | Moderate | MemGPT architecture |
| Agent memory ≠ LLM memory ≠ RAG (distinct concepts) | Strong | Memory in the Age of AI Agents (2025) |
| Zero context compression via periodic state consolidation | Moderate | InfiAgent (2026) |
